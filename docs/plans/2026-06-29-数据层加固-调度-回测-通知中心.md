## 0. 背景与原则

ChatGPT / Gemini 两份建议的共同结论：**数据源是命门，策略次要**。但通读现有代码后发现，本项目其实**已经把"数据层骨架"做掉了大半**——两份建议中相当一部分内容属于"已实现"或"技术栈不适用"（它们推荐的 AkShare/Tushare 是 Python 库，而本后端是纯 TS）。

因此本计划不另起炉灶，而是在现有架构上**加肉**，遵循三条原则：

1. **不引入 Python**。所有数据源继续用 TS `fetch` 直接打 HTTP 接口（AkShare 本质也是封装这些接口，直接复刻即可）。
2. **上层零改动**。所有增强都收敛在 `dataProvider` / provider 层之下，`Strategy` / `scan` / 前端接口不变。
3. **不追高频**。基金是收盘净值（日级），黄金做的是"买/持/卖"决策而非高频交易。现有 `TTL=1h` 合理，只补"关键时点强制刷新"，不做秒级轮询。

### 现状速览（已实现，无需重做）

| 能力 | 位置 | 状态 |
|---|---|---|
| 三级回退链 CSV > 在线 > 模拟 | `dataProvider.ts` `loadReal` | ✅ |
| 统一 Candle 标准化 | 所有 provider | ✅ |
| 缓存 + 1h TTL | `dataProvider.ts` `cache` | ✅ |
| 重试 + 退避 | `dataProvider.ts` `retry` | ✅ |
| 后台预热（间隔拉取防限流） | `warmUpAll` | ✅ |
| 数据源状态可查 | `/api/datasources` | ✅ |
| 历史回测（同档位胜率） | `backtest.ts` | ✅ 存在，但**未验证是否真在跑** |
| 通知中心 | — | ❌ 完全空白 |

## 1. 模块 A：数据源并列冗余（★★★）

### 问题
当前每个资产只有**一条在线源**：
- 黄金 → 东方财富 `push2his.eastmoney.com`（`providers/eastmoney.ts`）
- 基金 → 天天基金 `api.fund.eastmoney.com/f10/lsjz`（`providers/tiantian.ts`）

单点风险真实存在：这两个接口偶发限流、改字段、返回空。现在靠 `retry(3)` 硬撑，撑不住就掉到模拟数据——而模拟数据会让雷达**显示假信号**，这是最危险的（用户基于假信号下单）。

### 目标
给黄金、基金各加 1 个**并列冗余源**，主源失败时自动切备源，**主备都失败才回退模拟数据**，并把"用的是哪个源"暴露到 `/api/datasources`。

### 设计

**黄金备源：新浪财经实时报价**
- 接口：`https://hq.sinajs.cn/list=hf_XAU`（或 SGE 现货 `hq.sinajs.cn/list=hf_AU9999`，需实测可用变体）
- 新浪返回的是**实时最新价**（单点），不是历史 K 线。所以策略是：**历史 K 线仍由东方财富提供，新浪只用于"补最新一日 + 校验最新价"**，避免历史序列断层。
- 若东方财富 K 线整段失败 → 退回模拟（新浪单点无法重建历史序列）。

**基金备源：东方财富盘中估值接口**
- 接口：`https://fundgz.1234567.com.cn/js/{fundCode}.js`（返回 `jsonpgz({...})` 包裹的实时估值）
- 用途有二：
  1. 盘中（9:30–15:00）补一条**当日估值**作为最新 K 线，让雷达盘中就能动（当前只有收盘后才有当日净值）。
  2. 作为 `lsjz` 历史净值接口失败时的"最新价"兜底（同样无法重建历史，但至少最新价是真的）。

**实现方式：provider 列表化（最小侵入）**

改 `AssetConfig`（`types.ts` + `data/assets.ts`），把单源改成"主源 + 备源列表"：

```ts
// types.ts
export interface ProviderRef {
  kind: 'eastmoney_gold' | 'eastmoney_fund' | 'sina_gold' | 'eastmoney_fundgz';
  ident: string; // secid / fundCode / sina符号
}
export interface AssetConfig {
  // ...原有字段
  providers: ProviderRef[];      // 按优先级排列，主源在前
  // 保留 source/secid/fundCode 做向后兼容（派生自 providers[0]）
}
```

改 `dataProvider.ts` `loadReal`：把"按 source 单调用"改成"遍历 `providers`，依次尝试，第一个返回有效数据即用"。`retry` 仍套在最外层。

新增文件：
- `server/src/data/providers/sina.ts` — `fetchGoldSnapshot(symbol)` 返回最新价
- `server/src/data/providers/fundgz.ts` — `fetchFundEstimate(fundCode)` 返回盘中估值

**"最新价校验/补点"逻辑**（新函数 `enrichLatest`）：拉到历史 K 线后，再用备源拉一次最新价，若最新价日期晚于 K 线最后一根，追加一根当日 K 线（open=high=low=close=最新价，volume=0）。这步**可选开关**，避免盘中估值污染收盘净值序列。

### 改动文件
- `server/src/types.ts` — 加 `ProviderRef`，`AssetConfig` 加 `providers`
- `server/src/data/assets.ts` — 8 个标的补 `providers` 字段
- `server/src/data/dataProvider.ts` — `loadReal` 改多源遍历；新增 `enrichLatest`
- `server/src/data/providers/sina.ts`（新）
- `server/src/data/providers/fundgz.ts`（新）
- `getSourceStatus()` — 多返回一个 `usedProvider` 字段（实际命中的源）

### 风险
- 新浪 `hq.sinajs.cn` 需要 `Referer: https://finance.sina.com.cn` 才返回数据，否则 403。需实测。
- `fundgz` 盘中估值只在交易时段有值，非交易时段返回空——要容错。
- 模拟数据回退是**安全网而非正常路径**，目标是把回退率从"偶尔"压到"几乎不"。

### 验证
- 断网测主源失败 → 看日志是否切备源而非直接掉模拟。
- `/api/datasources` 返回的 `usedProvider` 与实际一致。
- 黄金/基金最新 K 线日期 = 当天（交易日内盘中则为估值日）。

## 2. 模块 B：交易日收盘后强制刷新（★★★★）

### 问题
现在缓存 `TTL = 1h`，意味着基金净值发布窗口（约 20:00–22:00）后，最坏要等 1 小时才看到当日新净值。对"每天看一眼"的雷达，这个延迟可接受但不优雅。

### 目标
- **交易日 15:30 后**强制失效基金缓存（净值开始陆续发布）。
- **交易日 22:00 后**再强制刷新一次（确保拿到当日最终净值）。
- 黄金因夜盘延续到次日凌晨 2:30，单独处理：交易日 15:30、20:00、02:30 各刷一次。
- **非交易日不刷新**，省接口调用、防限流。

### 设计
新增 `server/src/scheduler.ts`：

```ts
// 基于 setInterval 的轻量调度，不引第三方库
// 交易日判定：排除周末；节假日暂不处理（国内节假日表需手动维护或接入，
//   MVP 阶段接受节假日空跑——接口返回空数据不会污染缓存）
export function startScheduler(): void {
  // 每分钟检查一次时间表，命中则触发 refreshAsset
}
```

关键点：
- 复用现有 `refreshAsset`，不新写抓取逻辑。
- 刷新前主动 `cache.delete(id)` 让 `getAsset` 暂时回退模拟，抓成功后填回真实——避免显示过期缓存。
- 在 `index.ts` 启动时调用 `startScheduler()`。

**交易日历**：MVP 用"周一至周五"判定。若要精确，后续可加 `server/data/holidays.json` 手动维护当年节假日（国内节假日无稳定免费 API，手动维护一年一次成本可接受）。

### 改动文件
- `server/src/scheduler.ts`（新）
- `server/src/index.ts` — 启动时 `startScheduler()`

### 风险
- 节假日空跑：接口返回空，`retry` 抛错，缓存不被污染（安全）。日志会多几条 warn，可加节假日表消除。
- 时区：服务器时间需确认是 UTC+8（WSL 环境要核对 `date` 输出），否则收盘判定错位。

### 验证
- 手动改系统时间或 mock 时钟，观察 15:30/22:00 是否触发刷新。
- 看日志 `📡 [fund-xxx] 已加载真实数据 N 根K线` 出现时机。

## 3. 模块 C：回测验证（★★★★）

### 问题
`backtest.ts` 写得不错（同档位历史匹配 + 20 日胜率 + 人话 note），`scanOne` 也调了它并带 `backtestCache`。但有两个隐患没验证：

1. **是否真在跑？** `scanOne` 用 `getAssetSlice(id, days)` 给策略算当前信号，但回测用 `getAsset(id)!`（完整历史）。若某资产长期掉在模拟数据（只有 ~250 根），`candles.length < 60 + 20 + 10` 判不过，回测返回 `undefined`——**前端拿到的是 undefined 而非"样本不足"提示**，用户会以为没回测功能。
2. **信号到底有没有价值？** 现在回测只对"当前信号档位"算胜率，没有跨策略、跨资产的全局统计。无法回答"classic 策略整体靠不靠谱"。

### 目标
分两步：

**C1（验证现状）：写一个一次性脚本，跑出全局回测报告**
- 新增 `server/src/scripts/audit-backtest.ts`（`tsx` 直接跑）
- 对每个资产 × 每个策略，调 `backtestSignal`，输出表格：资产、策略、matched、winRate、avgReturn、note
- 顺便统计：哪些组合 `matched < 3`（样本不足）、哪些 `winRate < 50%`（信号反向、该删该改）
- 结论用于决定：哪些策略对哪些资产有效，哪些是噪音

**C2（小修）：把 `undefined` 兜成"样本不足"**
- `backtestSignal` 在 `candles.length` 不足时，当前返回 `undefined`。改成返回一个 `BacktestResult`，`note = '历史数据不足，无法回测'`，`matched=0`，避免前端静默丢失。
- 前端 `AssetDetail` 若有"无回测"分支，确认能正确渲染这个 note（可能需要看一眼前端，最小改动）。

**C3（可选，后续）：信号价值门槛**
- 若 C1 报告显示某策略对某资产 `winRate` 长期 < 50%，在 `scanAll` 给该信号加一个 `lowConfidence: true` 标记，前端徽章变灰，提示"历史胜率不高"。
- 这步依赖 C1 结果，**不在本次实施范围**，留作后续。

### 改动文件
- `server/src/scripts/audit-backtest.ts`（新，一次性脚本）
- `server/src/services/backtest.ts` — 数据不足时返回"样本不足"对象而非 undefined
- `server/src/types.ts` — `BacktestResult.matched` 已是 number，无需改；可能加 `sampleInsufficient?: boolean`
- `client/src/components/AssetDetail.tsx` — 检查回测渲染分支（最小改动）

### 风险
- 模拟数据下回测无意义（几何布朗运动必然趋势向上，胜率虚高）。C1 脚本必须**只在真实数据加载后跑**，或在输出里标注每个资产 `loaded` 来源，模拟数据的行单独分组、不纳入结论。
- `backtestCache` 在长生命周期下不会失效，若数据刷新后缓存仍指向旧回测——可加"刷新后清回测缓存"联动。

### 验证
- 跑 `tsx src/scripts/audit-backtest.ts`，看报告。
- 故意把某资产 `candles` 截短，确认详情页显示"历史数据不足"而非空白。

## 4. 模块 D：通知中心（★★★★★，最高用户价值）

### 问题
当前雷达是"被动看板"——你得记得去开网页。对"每天看一眼"的决策场景，真正有价值的是**信号变化时主动推给你**。

### 目标
- 每日固定时点（如交易日 22:00 刷新完净值后）扫描全部资产，对**信号发生档位变化**的标的（hold→buy、hold→sell、buy→sell 等）生成推送。
- 推送内容：资产名、新档位、综合分数、回测胜率（来自模块 C）、人话理由 top3。
- 多通道：MVP 至少接一种，架构上预留多通道。

### 设计

**通知触发：信号变化检测**
- 新增 `server/src/services/notify.ts`
- 维护一个 `lastSignalState: Map<assetId, action>`（进程内存即可，重启后从上次快照恢复——可选落盘 `server/data/signal-state.json`）
- 每次 `scanAll` 后比对：`prevAction !== currAction && currAction !== 'hold'` 才推（hold 不打扰）
- 防抖：同一档位变化 24 小时内不重复推

**通道抽象**
```ts
export interface Notifier {
  name: string;
  send(msg: Notification): Promise<void>;
}
export interface Notification {
  title: string;
  body: string;
  assetId: string;
  action: 'buy' | 'sell';
  score: number;
  winRate?: number;
}
```
- `CompositeNotifier` 串多个通道，任一失败不影响其他。

**通道候选（见下方开放问题，需你拍板）**：

| 通道 | 优点 | 缺点 | 实现成本 |
|---|---|---|---|
| **Server酱（推荐 MVP）** | 一行 HTTP POST 推到微信，零部署 | 免费限频 5 条/天，需注册拿 sendkey | 极低 |
| **Telegram Bot** | 无限频、支持富文本 | 国内需代理 | 低 |
| **邮件（SMTP）** | 无频率限制、可带图 | 需 SMTP 配置，到达慢 | 低-中 |
| **钉钉/飞书 webhook** | 企业群直接收 | 需建群 | 低 |

配置全部走环境变量（`.env`，需加 `dotenv` 依赖），不硬编码。

**调度联动**：通知检查挂在模块 B 的 `scheduler` 上——22:00 刷新完净值后触发一次 `scanAndNotify()`，盘中（15:30）可选触发一次（用估值，标注"盘中初步信号"）。

### 改动文件
- `server/src/services/notify.ts`（新）— 触发逻辑 + 通道抽象 + `CompositeNotifier`
- `server/src/notifiers/serverchan.ts`（新，或其他通道）
- `server/src/scheduler.ts` — 加 `scanAndNotify` 调用
- `server/src/types.ts` — 加 `Notification` / `Notifier` 接口
- `server/src/index.ts` — 启动时初始化通知器
- `server/package.json` — 加 `dotenv`（若用 .env）
- `server/.env.example`（新）— 示例配置

### 风险
- Server酱免费 5 条/天：8 个标的若同日齐变会超限。需做"按分数排序只推 top3"或升级付费。
- 信号频繁抖动（buy↔hold 反复）会骚扰。防抖 + 只推非 hold 是底线。
- `lastSignalState` 进程内存重启丢失 → 重启后会重推一遍。落盘 `signal-state.json` 可解决。

### 验证
- 手动改某资产 `lastSignalState` 触发推送，看微信/邮件是否收到。
- 断开通道（假 sendkey），确认主流程不崩、只 warn。

## 5. 推荐实施顺序

依赖关系：B 依赖 A 的多源刷新能力（刷新更稳才有意义）；D 依赖 B 的调度；C 独立。

```
A 数据源冗余  ──▶  B 调度刷新  ──▶  D 通知中心
                                         ▲
C 回测验证  ──────────────────────────────┘（通知里带胜率）
```

建议顺序：**A → C → B → D**

- A 先做：让数据先稳，后续都受益。
- C 紧跟：在加调度前先知道"现有信号到底有没有价值"——若 C1 报告显示某策略胜率全 < 50%，那 D 推送的就是垃圾信号，不如先调策略。
- B 第三：有了 A 的稳定数据 + C 的信号价值判断，调度刷新才有的放矢。
- D 最后：是最终交付层，把前三块的成果推出去。

每个模块独立可验证、可单独提交。

## 6. 开放问题（需你拍板后再进入实施）

1. **通知通道选哪个做 MVP？**（见模块 D 表格，我推荐 Server酱 → 微信，成本最低）
2. **盘中估值要不要纳入？**（模块 A 的 `enrichLatest`）：纳入则雷达盘中能动，但盘中估值 ≠ 收盘净值，信号可能盘中/盘后不一致。倾向"纳入但标注"。
3. **节假日历要不要现在就做？**（模块 B）：MVP 建议不做，接受空跑；要做则需手动维护 `holidays.json`。
4. **`lastSignalState` 要不要落盘？**（模块 D）：落盘可防重启重推，但多一个文件要维护。建议落盘。
5. **模拟数据回退要不要在前端显著标红？** 当前 `/api/datasources` 能查，但雷达主页本身不标。强烈建议加——基于模拟数据下单是最大风险。
